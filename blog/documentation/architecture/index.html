Design / Architecture

Documentation 1 Sept 2021

This is usually known as the Software Design Document (SDD). It’s a written description of a software product that gives a development team overall guidance to the architecture of the software project. It should have the following key sections:

Introduction

This is the first section, though is often behind a table of contents. It summarizes the document and should contain the following information:

  • purposes
  • scope
  • intended audience
  • design goals and rationale
  • definitions of important terms or reference glossary
  • assumptions and dependencies (such as being able to send and receive texts)
  • constraints (such as must work on Windows 7)
  • goals and guidelines
  • development methods (such as the MVC pattern for high cohesion and low coupling)

System Architecture

This is a high level overview of functionality and responsibilities of the system. It should describe high level components and how they work together. It doesn’t have to only contain descriptive text, it could include diagrams, flowcharts, models, scenarios, and high level use cases showing system behaviour and structure.

Detailed System Design

This section should describe in detail the functionality and responsibilities for each component of the system. It often uses class and sequence diagrams to show relationships and flows. It can include:

  • design patterns and techniques
  • modules, classes, files, etc.
  • data structures
  • algorithms
  • interfaces

Data Design

This describes the data structures, databases, and storage units and their relationships. This section will often contain an entity relationship diagram (ERD) to show these structures and relationships. This section can include:

  • frequency of updates
  • security requirements
  • recovery
  • report generation
  • interfacing systems

User Interface

This section describes the functionality from the user’s perspective. It will often include wireframes or mock-ups of what the UI will look like. It describes the controls and their behaviours, as well as supported orientations and dimensions.

Glossary / Appendix

These are the final sections and only need to be included if relevant.
