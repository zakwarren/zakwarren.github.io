Development Documentation

Documentation 2 Sept 2021

There are a number of places that low level development documentation resides. These have their own specific purposes, but all aim to provide developers with information about design choices.

Code Comments

As developers, we usually strive for self describing code. However, comments still have their place and can be particularly useful if returning to a codebase after a long time or for developers new to a project. Good comments are:

  • high level
  • useful at the class level
  • used to identify specific points (TODO, HACK, and UNDONE), which can be used by your IDE to generate a task list

We shouldn’t use comments to:

  • state something obvious or redundant
  • explain poorly written or poorly designed codebase
  • soft delete code

API Documentation

API documentation is one of the most useful pieces of documentation for a developer. If you’ve been a developer for any length of time, you’ll have had experiences with great documentation and bad or no documentation when trying to pick up new APIs. Good API documentation will include:

  • overview and concepts – narrative description of different parts of the API and how they interface
  • reference documentation – descriptions of functions and objects in the API
  • installation and getting started
  • Tutorials and training
  • troubleshooting
  • SDK tools documentation – if you have it, then how to install, use, compile, and deploy
  • license information

Unsurprisingly, the audience for API documentation is developers. However, these can be broken down into a few categories:

  • developers new to the API
  • developers needing to debug
  • business people or developers evaluating the API

Readme files

A readme file is a file, usually txt or md, that helps users or other developers know how to do things with your software. We generally write the readme file before our code, or very early on in the coding process. Remember, until you’ve written about your software, you have no idea what you’ll be making.

A readme file should include:

  • software name (and could include a version number)
  • date of release
  • short description of the software
  • installation requirements and instructions
  • copyright and licensing information
  • contact information for the developer or distributor

If it makes sense, you could also include:

  • file manifest (list of files included in the package)
  • configuration instructions
  • operating instructions
  • known bugs
  • troubleshooting
  • credits and acknowledgements
  • change log (usually for developers)
  • news or updates (usually for users)
