End-User Documentation

Documentation 4 Sept 2021

End-user documentation may be the most important type of documentation that you’ll create with a software project. It’s aimed at teaching the users how to use the software and to help them if they get stuck.

User Guide

This helps the user to understand and use a system. It has several key sections:

Overview

This often follows a table of contents. It’s a high level description of the product and describes it’s major features, the benefits a user gets from using the software, and, if applicable, how it fits in with other products.

Getting Started

This is where you should describe the setup and configuration information. It’s best to do this in a task-based style, so that the user can start using the system quickly. It’s also a good place to list the prerequisites for using the system.

Tutorials

This should be organised by main features or tasks.

Troubleshooting

This section details possible errors or problems, and describes how to resolve them.

Frequently Asked Questions (FAQs)

This section can list common questions that users have, along with answers and references to additional information.

Support

This section should detail how to get additional help and the contact information for any support desks.

Glossary and Index

These are the final sections and only need to be included if relevant.

Quick Reference Guide

The quick reference guide should distil the key information from the user guide into a short, easy to understand document. It can even be a single page.

You shouldn’t use walls of text in the quick reference guide. Instead, you can include:

  • diagrams
  • reference tables
  • steps
  • graphics
  • infographics

Release Notes

Release notes document the differences between two versions of the same software, such as v1 and v2. It should describe:

  • release date
  • product versions
  • previous product version
  • the benefits of upgrading
  • new, cool features
  • bug fixes

New Features

This should list the new features or functionality in this release, and describe the benefits of the new features.

Enhancements

This should list changes to existing features, include the justification for the change, and describe how the change is beneficial.

Resolved Issues

This should detail any problems that existed in the previous release that were fixed in this release. It should include a brief description of the solution to the issue.

Known Issues

This should list any bugs that are known to exist when the software releases. This lets the client determine if they will be an issue for them. It establishes credibility, but make sure that they are resolved in the next release.

Frequently Asked Questions (FAQs)

This should include common or anticipated user questions and their answers.

Optional Sections

You could include the system requirements and installation instructions in the release notes, but only if these have changed since the last release.
